# AGENTS.md — guide for AI agents working with `contoso-dbt`

This document explains how to navigate, query, and reason about the data in
this pipeline. Read it before doing any analysis, ad-hoc queries, or doc
generation against the Contoso datasets.

## What's in this repo

Two pipelines materialize the same Contoso retail data product, side-by-side,
into different BigQuery datasets so they can be compared:

| Pipeline | Tooling | Datasets (in `bruin-playground-arsalan`) |
|---|---|---|
| `contoso/` | Bruin (Python ingest + SQL transforms) | `contoso_raw` / `contoso_staging` / `contoso_reports` |
| `contoso-dbt/` (this dir) | dlt (Python ingest) + dbt (SQL transforms) | `contoso_dbt_raw` / `contoso_dbt_staging` / `contoso_dbt_reports` |

Both have the same shape: **23 raw tables → 10 staging models → 7 report models**.
Naming is parallel: `sales_fact` (bruin) ↔ `stg_sales_fact` (dbt).

## The context layer (`context/`)

`context/assets/` is the agent's primary entry point for understanding what's
in the warehouse. Each materialized table has one `.asset.yml` containing:

- `name` — fully-qualified `dataset.table` (e.g. `contoso_dbt_staging.stg_sales_fact`)
- `description` — multi-paragraph business context (purpose, grain, lineage,
  typical use cases, row count, size)
- `tags` — domain / layer / sensitivity tags
  (`domain:retail`, `layer:staging`, `sensitivity:pii`, …)
- `columns[]` — every column with `type`, `description`, and quality `checks`
  (`not_null`, `unique`, `accepted_values`, …)

These YAMLs are **documentation only** — no SQL is materialized from them.
They're the AI-readable counterpart to dbt's `schema.yml`, generated by
`bruin import database` + `bruin ai enhance` against the live BigQuery
datasets.

**To regenerate after schema changes:** `./contoso-dbt/generate_context.sh`
(takes several minutes — runs `bruin ai enhance --claude` on every asset).
Use `--skip-enhance` for a fast import-only refresh, or `--skip-import` to
re-enhance existing YAMLs.

## How agents should work with this data

### 1. Start by reading context, not by querying

Before issuing any BigQuery query, read the relevant `.asset.yml` to learn:
- the table's grain (one row per *what*?)
- which columns carry the business logic
- which columns have quality guarantees (`not_null` / `unique` checks)
- whether the table is PII-sensitive (`sensitivity:pii` tag)

Cheap, no-cost, and avoids confidently-wrong queries against unfamiliar columns.

### 2. Use the Bruin MCP for *Bruin tooling* questions

The Bruin MCP server exposes:

- `bruin_get_overview` — high-level tour of Bruin's capabilities
- `bruin_get_docs_tree` — table of contents for Bruin docs
- `bruin_get_doc_content` — fetch a specific doc page

Use these when you need to know **how Bruin works** (asset types, connection
config, `ai enhance` flags, etc.) — *not* for project-specific data questions.
Project-specific knowledge lives in `context/assets/*.asset.yml`.

### 3. Use the Bruin CLI to run queries

The CLI is the easiest path to actual rows. From the repo root:

```bash
# One-off SQL (uses connection from context/.bruin.yml)
bruin query \
  --config-file contoso-dbt/context/.bruin.yml \
  --connection contoso_dbt_bq \
  --query "SELECT category_name, SUM(revenue_usd) AS rev
           FROM \`bruin-playground-arsalan.contoso_dbt_reports.rpt_revenue_by_segment\`
           WHERE year = 2024
           GROUP BY 1 ORDER BY rev DESC LIMIT 10"

# Validate the context pipeline structure
bruin validate --config-file contoso-dbt/context/.bruin.yml contoso-dbt/context

# Re-enhance a single asset's docs (rarely needed)
bruin ai enhance --claude contoso-dbt/context/assets/contoso_dbt_reports/rpt_profit_and_loss.asset.yml
```

The isolated `context/.bruin.yml` uses Google Application Default Credentials
(`gcloud auth application-default login`) so no keyfile is required. The repo
root has its own `.bruin.yml` for the bruin pipeline — **always pass
`--config-file contoso-dbt/context/.bruin.yml`** when working in the dbt
context, otherwise Bruin loads stale connection definitions.

### 4. Parity-check against the Bruin pipeline when a number looks off

Both pipelines materialize the same data, so cross-dataset diffs catch bugs:

```sql
SELECT 'bruin' AS pipe, SUM(revenue_usd) AS rev
FROM `bruin-playground-arsalan.contoso_staging.sales_fact`
UNION ALL
SELECT 'dbt',           SUM(revenue_usd)
FROM `bruin-playground-arsalan.contoso_dbt_staging.stg_sales_fact`;
```

A non-trivial divergence (>1¢ on aggregate metrics) means one side has a bug.
Tiny float-rounding differences on `AVG(...)` over millions of rows are
expected.

## Common agent recipes

**"Summarize a table"** → read `context/assets/<schema>/<table>.asset.yml`,
then quote the description + grain + tags. No query needed.

**"Find tables related to X"** → grep tags/descriptions across
`context/assets/`:
```bash
grep -lE "tag.*(marketing|campaign)" contoso-dbt/context/assets/**/*.yml
```

**"What columns are PII?"** → grep for `sensitivity:pii` in the tags, then
read the column list of those assets.

**"Run analysis Q"** → (a) read the relevant assets to pick the right grain,
(b) use `bruin query` (or any BigQuery client) with ADC, (c) cite the
asset(s) you read in the answer.

**"Schema changed — refresh docs"** → `./contoso-dbt/run_pipeline.sh` then
`./contoso-dbt/generate_context.sh`.

## Things to avoid

- **Don't query without reading the asset YAML first.** Hallucinated column
  names waste cycles.
- **Don't edit `context/assets/*.asset.yml` by hand.** They're regenerable
  artifacts. Improve them by re-running `bruin ai enhance`, or update the
  upstream dbt model + its `schema.yml` (which feeds the next regeneration
  via `bruin import database`).
- **Don't mix the dbt context with the bruin pipeline's `.bruin.yml`.** The
  repo-root config is for `contoso/`. Always scope to
  `contoso-dbt/context/.bruin.yml` when in this pipeline.
- **Don't assume row counts in descriptions are live.** They reflect the
  state at last `generate_context.sh` run. For current counts, query
  `INFORMATION_SCHEMA.PARTITIONS` or `COUNT(*)`.
